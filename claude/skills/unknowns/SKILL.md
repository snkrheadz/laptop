# Unknowns Discovery Skill

Anthropic Thariqの "A Field Guide to Fable" で紹介されたunknownsフレームワーク
（Known/Unknown Knowns/Unknowns）に基づき、着手前に認識のズレを潰すためのスキル。
**Blind Spot Pass → Interview** の2段階で進める。

## 引数

`$ARGUMENTS`: これから取り組むタスクの説明
（例: "新しい認証プロバイダを追加したいが、authモジュールを知らない"）。
空の場合はユーザーに何に取り組むか尋ねる。

## Step 1: Blind Spot Pass（未知の未知 → 既知の未知への変換）

目的: ユーザーが気づいていない前提・過去の経緯・非自明な制約を先に晒す。

1. `$ARGUMENTS` から対象領域を特定する。
2. codegraph_explore / Explore agent / `git log` などでその領域を調査する。見るべき観点:
   - 既存の設計パターンや、過去のバグ修正・リバートの背景（"なぜこうなっているか"）
   - 一見自明に見えて実は罠がある箇所（命名の衝突、隠れた依存、非対称な挙動）
   - 一般的なベストプラクティスとこのコードベース固有の流儀が食い違う点
3. 調査結果を「盲点になりそうな点」として箇条書きで提示する。各項目は1〜2文、
   なぜ盲点になりうるかを添える。
4. **この時点では実装にもインタビューにも進まない。** ユーザーの反応
   （「知ってた」「知らなかった」「それは関係ない」）を待つ。

## Step 2: Interview（既知の未知を潰す）

Blind Spot Passへの反応を踏まえ、残っている曖昧さを一問一答で確認する。

- 一度に1問だけ聞く。まとめて質問リストを投げない。
- **回答によってアーキテクチャ/実装方針が変わる質問を優先する。** 些末な質問は
  後回しにするか省略する。
- 選択肢が明確な場合はAskUserQuestionツールを使う。オープンな設計判断は
  テキストで聞く。
- ユーザーが「もう十分」「進めて」と言うか、高価値な曖昧さがなくなるまで続ける。

## Step 3: 合意の要約

インタビュー後、以下を簡潔に提示する。

- 分かったこと（Known Knowns）の要約
- まだ残っている前提（あれば明示し、「この前提で進める」と宣言する）

破壊的変更・スキーマ変更・複数サービス跨ぎなど非自明な変更であればここで
`EnterPlanMode` へ、それ以外は要約後にそのまま実装に進む。

## 使用例

```
/unknowns 新しい認証プロバイダを追加したい。authモジュールは初見。
/unknowns このリポジトリのCI設定を変えたいが、既存の意図が分からない。
```
