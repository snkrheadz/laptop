> 注: 本件はドメインアプリではなく CI/シェル検証ツーリングの変更。DDD/レイヤの語彙は無理に当てはめず、該当範囲のみ honest に記述する。

## ドメインモデル（DDD）
ユビキタス言語のみ定義する（エンティティ/集約は持たない性質の変更）。
- **シェルスクリプト（対象）**: git 管理下のファイルのうち、`shellcheck` が解析可能なシェルで書かれたもの。判定は「拡張子 `.sh`」**または**「先頭行 shebang が `sh`/`bash`/`dash`/`ksh` 系」。
- **検査対象集合**: 上記の判定で動的に導出される集合。**人手のリストではなく導出結果**であることが本質。
- **検証ハーネス**: その集合に対し `shellcheck` を実行し、1件でも指摘があれば失敗を返す手続き。CI と（任意で）ローカルが同一の手続きを共有する。

## レイヤ構成（クリーンアーキテクチャ）
依存方向は「CI 設定 → 検証スクリプト → shellcheck」の一方向（外→内）。CI 定義は検証ロジックの詳細を知らず、単一スクリプトを呼ぶだけにする。
- **infra / 起動点**: `.github/workflows/main.yml` の Shellcheck ステップ。対象列挙の方法を一切持たず、検証スクリプトを呼ぶだけ。
- **interface-adapter / 検証手続き**: 新規 `scripts/lint-shell.sh`。git 管理下から対象集合を導出し、`shellcheck` に渡し、結果を終了コードに集約する。**唯一の対象判定ロジックの所在**（CI も人間もエージェントもここを通る = 一貫性）。
- **外部ツール**: `shellcheck`（既存依存、新規導入なし）。
依存ルール: CI（外）はスクリプト（内）に依存するが逆はなし。スクリプトは shellcheck の有無に依存するのみ。

## サービス境界 / 統合点
- 単一リポジトリ内のツーリング変更。サービス分割なし（モノリポのまま正しい）。
- 統合点は2つ: ①CI（`main.yml`）→ 検証スクリプト、②（将来）ローカル実行 → 同一スクリプト。同一スクリプトを通すことで「CIとローカルの検査内容が一致」を構造的に保証する。
- **既存 PostToolUse hook `validate-shell.sh` とは境界を分ける**。あれは「編集された1ファイルを即時検査」するイベント駆動の別関心事。本件は「全体ゲート」。両者は統合しない（hook 側の `.sh` 限定は別バックログ。下記リスク参照）。

## 主要な設計判断
- 判断: 対象列挙を**専用スクリプト `scripts/lint-shell.sh` に集約**し、CI はそれを呼ぶだけにする / 理由: 判定ロジックが1箇所に集まり、CI とローカルが同一手続きを共有できる（=「CIと一致」を構造で担保）。スクリプト自身も `.sh` なので自分自身の検査対象に入る（自己被覆）/ 却下案: `main.yml` に列挙ロジックをインラインで埋め込む → CI 専用になりローカル再現不可、YAML 内シェルは検査もされず脆い。
- 判断: 対象判定を**「拡張子 `.sh` OR シェル shebang」のハイブリッド**にする / 理由: requirement の確定事項（shebang 基準で `bin/tat`・将来の拡張子なしも取り込む）。`.sh` も併用するのは、shebang を持たない/将来持たない `.sh` を確実に拾うため / 却下案: 拡張子のみ → `bin/tat` を取りこぼす（本件の趣旨に反する）。shebang のみ → shebang 欠落の `.sh` を取りこぼす可能性。
- 判断: 対象の母集合は **`git ls-files`（git 管理下）** とする / 理由: requirement の「リポジトリ管理下のすべて」に一致。`.git/` や未追跡の生成物・ベンダーディレクトリを自然に除外できる / 却下案: `find .` 全走査 → 管理外ファイルや生成物まで拾い、除外管理が増える。
- 判断: `shellcheck` が解析できる shebang のみ対象（`sh`/`bash`/`dash`/`ksh`）、**`zsh` は対象外** / 理由: `shellcheck` は zsh を解析できずエラーになる。zsh 設定は別の検証関心事 / 却下案: 全 shebang を一律対象 → zsh ファイルで誤検知・CI 恒常失敗。
- 判断: 恒久除外は設けない。ただし**除外が必要になった場合の記録口を1箇所**（スクリプト内の明示された除外指定）に用意し、レビュー可能にする / 理由: requirement AC4（除外は明示的・レビュー可能）を将来も満たすため。現時点では空 / 却下案: 除外機構を持たない → 将来 AC4 を満たせなくなる。

## 既存コードへの影響（blast radius）
- **`.github/workflows/main.yml`（変更）**: Shellcheck ステップのコマンドを、ハードコードされた `shellcheck install.sh rollback.sh scripts/*.sh` から、新規検証スクリプトの呼び出しに置換。gitleaks ステップ・トリガー・権限は不変。
- **`scripts/lint-shell.sh`（新規）**: 対象列挙 + `shellcheck` 実行 + 終了コード集約。`scripts/*.sh` に属するため、CI の従来パターンでも新パターンでも検査対象に入る。
- **影響を受けないが関連**: `claude/hooks/validate-shell.sh`（PostToolUse hook、別関心事・無変更）、`bin/tat`（新たに検査対象に**入る**側。現時点で shellcheck 通過済みと実測）。
- 破壊的変更なし。CI の他ジョブ・ローカル開発フローに後方非互換なし。現状 10 本すべて shellcheck 緑（実測）なので、対象拡大で新たな CI 失敗は出ない前提。

## 受け入れ条件 → 設計 の対応表
| requirement の受け入れ条件 | それを満たす設計要素 |
| --- | --- |
| CI 起動時に全シェルスクリプト（hooks・statusline 含む）を検査 | `lint-shell.sh` が `git ls-files` から `.sh`＋shell shebang を導出 → `main.yml` がそれを呼ぶ。hooks/statusline/bin/tat が母集合に含まれる |
| 不備があれば CI を失敗させる | `lint-shell.sh` が shellcheck 指摘を終了コード非0に集約 → CI ステップが失敗 |
| 新規スクリプトを手動登録なしで対象化 | 対象は `git ls-files` + 判定からの**動的導出**。ハードコードリストを廃止 |
| 除外は明示的・レビュー可能に記録（必要時） | `lint-shell.sh` 内に明示された除外指定の口を1箇所用意（現時点は空）。差分でレビュー可能 |

## リスク / 未確定
- **shebang 判定の頑健性**: `#!/usr/bin/env bash` 等の env 経由、引数付き shebang、CRLF、BOM を取りこぼさないこと。実装時に実ファイルで検出結果が現状10本（9 .sh + bin/tat）と一致することを確認する（requirement の検証方法に対応）。
- **zsh ファイルの扱い**: 本設計は zsh を対象外とする。`zsh/configs/*.zsh` 等は別の検証関心事として残る（スコープ外）。
- **隣接ギャップ（スコープ外・無変更）**: PostToolUse hook `validate-shell.sh` は `.sh` のみ検査するため、`bin/tat` を編集しても hook は発火しない。CI 側は本件で被覆されるが、hook 側の被覆は別バックログ。ここでは指摘のみ。
- **検査内容は不変**: 本件は対象範囲のみ拡大し、shellcheck のオプション/重大度閾値は既存踏襲。新ルール追加はしない（requirement のスコープ out）。
