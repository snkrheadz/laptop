## 背景 / 解くべき課題
このリポジトリの中核スクリプト（インストールとロールバック）は、ホーム配下のファイルを退避し symlink を張り替える**破壊的操作**を含む。にもかかわらず、その振る舞いを検証する自動テストが一切なく、回帰（symlink を張り損ねる・既存ファイルを誤って壊す・ロールバックで復元できない等）を観測する手段がない。静的検査（shellcheck）は構文・規約の問題は捉えるが「実際に何が起きるか」は捉えない。スキルによる健全性チェックは対話的・事後的で、コミット/CI のゲートにならない。結果として、これら最も壊れると影響の大きいスクリプトが、検証ループの中で最も検証されていない。本要件は、利用者の実環境を一切汚さずに安全に実行できる振る舞いテストを用意し、インストール/ロールバック/symlink 作成の主要な振る舞いを観測可能にする。

## スコープ
- in: インストール・ロールバック・symlink 作成の主要な振る舞いを、隔離された環境に対して実行して検証する自動テスト。
- in: 検証対象の振る舞い = (1) 期待される symlink がリポジトリ内の正しい対象を指して作成される、(2) 既存ファイルが上書き前に退避（バックアップ）される、(3) ロールバックがバックアップから元の状態へ復元する、(4) 再実行しても壊れない（冪等）。
- in: テストが利用者の実ホーム環境・実パッケージ群（Homebrew/mise 等）を一切変更しないこと。
- in: テストを単一コマンドで実行でき、失敗時にどの振る舞いが壊れたか判別できること。
- out: パッケージインストール（Homebrew/mise/Xcode CLI 等）そのものの実行・検証。これらはネットワーク・実環境依存で、本テストの対象外（テスト中は実行しない）。
- out: 既存スクリプトの仕様変更・リファクタリング（テスト可能化のために最小限必要な調整は許容するが、機能変更はしない。詳細は設計で判断）。
- out: 新しい振る舞いの追加。既存の振る舞いを検証するのみ。

## 受け入れ条件（EARS）
- WHEN 振る舞いテストを実行したとき THE SYSTEM SHALL 隔離された環境に対してインストール処理の symlink 作成部分を実行し、期待される symlink がリポジトリ内の正しい対象を指して存在することを確認する。
- WHEN 既存ファイルが存在する隔離環境でインストール処理を実行したとき THE SYSTEM SHALL その既存ファイルが上書き前に退避（バックアップ）されていることを確認する。
- WHEN ロールバック処理を隔離環境で実行したとき THE SYSTEM SHALL バックアップから元の状態へ復元されることを確認する。
- WHEN インストール処理を隔離環境で2回実行したとき THE SYSTEM SHALL 2回目も失敗せず、結果が1回目と同じ健全な状態になること（冪等）を確認する。
- IF いずれかの振る舞いが期待と異なるとき THEN THE SYSTEM SHALL テストを失敗（非0終了）させ、どの振る舞いが壊れたか出力する。
- WHILE テストが実行されている間 THE SYSTEM SHALL 利用者の実ホーム環境・実パッケージ群を一切変更しない。

## 非機能要件 / 制約
- テストは利用者の実環境（実ホーム・実 Homebrew/mise 等）に副作用を一切持たないこと（最優先・安全要件）。
- テストはネットワークアクセスや管理者権限を要しないこと（パッケージインストールを実行しないため）。
- 新たな重いテストフレームワーク依存を持ち込まないこと（既存のシェル環境で完結することが望ましい）。
- 既存スクリプトの利用者向け挙動を変えないこと。テスト可能化のための調整は、通常実行時の振る舞いを保つ範囲に限る。
- テストの実行時間が実用上問題とならないこと。

## 検証方法
- テストを健全な状態で1回実行し、全ケース（symlink 作成・バックアップ・ロールバック・冪等）が成功し、全体が成功（0終了）することを確認する。
- 検証対象スクリプトに振る舞いの回帰を1つ仕込み（例: symlink を1つ張らない）、テストが失敗（非0）して該当ケースが落ちたと判別できることを確認する（確認後に戻す）。
- テスト実行の前後で、利用者の実ホーム配下（実際の symlink・バックアップディレクトリ等）が変化していないことを確認する。
- テストが単一コマンドで起動でき、ローカルと CI の双方で実行可能であることを確認する。

## 解決済みの前提（Gate ① レビューで確定）
- **テスト可能化のためのスクリプト最小調整を許容する。** install/rollback スクリプトに「直接実行時のみ一括処理（main）を走らせ、テストからは個別処理（symlink/backup/rollback）だけを呼べる」ガードを追加してよい。**通常実行時の利用者向け振る舞いは不変**であること（直接実行すれば従来通り全処理が走る）を条件とする。これは out「機能変更はしない」の範囲内（呼び出し可能性の調整であり機能の追加・変更ではない）。
- **テストは CI でも実行する。** 隔離環境で実パッケージに触れないため CI 実行は安全。コミット前検査／CI のゲートに振る舞いテストを組み込み、回帰を自動検出できる状態にする（柱A=テスト自動化の実体）。具体的な統合点（verify.sh への組み込み等）は設計で扱う。
- **「実ホームを使わない」は不変の安全制約。** 隔離環境の具体的な与え方（一時ディレクトリをホーム相当として渡す等）は設計フェーズで決めるが、実ホーム・実パッケージを一切変更しないことは要件として固定。
